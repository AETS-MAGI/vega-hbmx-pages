1) MIOpen solver selection — candidate list + IsApplicable filter 1) MIOpen solver 選択メカニズム — 候補列挙 + IsApplicable フィルタ

  • MIOpen does not use an explicit if-else fallback chain. Instead, it maintains a global solver registry and filters it through IsApplicable() at runtime. MIOpen は明示的な if-else フォールバックチェーンを使わない。代わりに、グローバルな solver レジストリを保持し、実行時に IsApplicable() でフィルタリングする方式。
// miopen/src/conv/solver_finders.cpp : 98-110 if(!parameters.use_winograd_only && !env::disabled(MIOPEN_DEBUG_CONV_IMPLICIT_GEMM)) { FindAllImplicitGemmSolutions(...) // FWD / BwdData FindImplicitGemmWrWAllSolutions(...) // BwdWrW } // miopen/src/include/miopen/find_solution.hpp : 326 / 381 / 451 else if(!solver.IsApplicable(ctx, problem)) { log("Not applicable"); continue; } // miopen/src/problem.cpp : 573 / 625 if(!solver->IsApplicable(ctx, problem_description)) log("Not applicable");
Design implication: Old-generation solver entries remain registered even as new solvers are added. On gfx900, newer solvers fall out via IsApplicable, and legacy solvers with explicit gfx900 whitelisting survive the filter — this is how the "maintain" layer works. 設計上の含意: 新しい solver が追加されても旧世代 solver のエントリはレジストリに残る。gfx900 では新しめの solver が IsApplicable で落ち、gfx900 を明示的に許可している旧世代 solver が生き残る構造 — これが「維持」層の仕組み。

5) XDLops solver family — systematically excluded for gfx900 5) XDLops solver ファミリー — gfx900 で系統的に除外

Runtime failures observed (forced execution):
• ConvHipImplicitGemmFwdXdlops / GrpFwd → std::vector::operator[] assertion abort (EXIT=134) — same across INT8, FP16, BFP16.

8) rocBLAS — gfx900 lazy loading + multi-stage runtime fallback 8) rocBLAS — gfx900 lazy loading + 多段実行時フォールバック

// rocblas/library/src/tensile_host.cpp // :232-240 getLazyLoadingArch — explicit gfx900 map if(deviceString.find("gfx900") != std::string::npos) return Tensile::LazyLoadingInit::gfx900; // :1232 hipBLASLt failure → Tensile fallback log("hipBlasLT failed, falling back to tensile."); // :1161 XF32 solution not found → FP32 fallback log("No Tensile solution found for XF32, fall back to FP32"); // rocblas/CMakeLists.txt : 80-85 # TARGET_LIST_ROCM_5.6 … TARGET_LIST_ROCM_7.1 all include "gfx900"
Summary of the "supplement" layer:
① The build system keeps gfx900 in target lists through at least ROCm 7.1.
② At runtime, gfx900 is explicitly routed to its own lazy-loaded library entry.
③ If hipBLASLt fails, rocBLAS automatically falls back to Tensile.
④ If no XF32 Tensile solution is found, rocBLAS falls back to FP32.
These are independent automatic recovery mechanisms — each one ensures a computation result even when a faster path is unavailable. 「補充」層のまとめ:
① ビルドシステムは ROCm 7.1 以降もターゲットリストに gfx900 を保持。
② 実行時に gfx900 は専用の lazy loading ライブラリエントリへ明示ルーティング。
③ hipBLASLt が失敗すると、rocBLAS は自動的に Tensile にフォールバック。
④ XF32 の Tensile 解が見つからない場合、rocBLAS は FP32 にフォールバック。
それぞれ独立した自動回復メカニズムで、高速経路が使えない場合でも計算結果を保証する。

Overall verdict 総合判定

The gfx900 (Vega10) computation path in the ROCm library stack is structured across four layers:

Maintain (build): gfx900 is present in rocBLAS CMake target lists through at least ROCm 7.1. The Tensile shared library is built with gfx900 support by default.

Manage (selection): MIOpen's IsApplicable filter system allows old-generation solvers to survive in the global registry. On gfx900, MLIR iGEMM and all XDLops-family solvers fall out, while Winograd-family, ASM 1×1, implicit GEMM v4r1 (HIP/ASM), and legacy direct ASM solvers remain available. Solver choice is also dtype-sensitive (FP32 / FP16 / BFP16 each route to different families).

Supplement (fallback): rocBLAS carries multi-stage runtime fallbacks. CK carries a dot4-absent sequential accumulation path. Tensile provides per-ISA capability gating and gfx900-specific lazy-load catalogs.

Distribute (shipped artifacts): The ROCm 7.2 package ships gfx900-targeted pre-compiled kernels (128 rocBLAS files) and architecture-specific tuning data (169K lines of MIOpen Perf DB). In these metrics, gfx900 surpasses RDNA 3 (gfx1100: 96 rocBLAS files, no Perf DB) and RDNA 2 (gfx1030: 88 rocBLAS files, 111K Perf DB lines). This is not residual code — it is the output of an active build pipeline.

2026-03-14 follow-up: A local Debug MIOpen build succeeded once the source tree was moved off CIFS to NVMe-local storage and the incompatible MLIR/CK pieces were disabled for the rebuild. Forced -S ConvMlirIgemmFwd can bypass the normal applicability gate, but then falls into incomplete downstream support (MIIR_INVALID_PARAM and/or missing gfx900 Perf DB records).

Confirmed gap: No natural INT8 solver selection beyond ConvDirectNaiveConvFwd was observed. The naive kernel operates via element-wise multiply-accumulate without dot-product instructions — functional but not peak-performance. ROCm ライブラリスタックにおける gfx900（Vega10）計算経路は 4 層で構成される:

維持（build）: gfx900 は少なくとも ROCm 7.1 まで rocBLAS CMake ターゲットリストに存在する。Tensile 共有ライブラリもデフォルトで gfx900 サポートを含む。

管理（selection）: MIOpen の IsApplicable フィルタ方式は旧世代 solver がグローバルレジストリに残ることを許す。gfx900 では MLIR iGEMM と全 XDLops 系 solver が落ちる一方、Winograd 系・ASM 1×1・implicit GEMM v4r1（HIP/ASM）・旧式直接 ASM solver が残る。dtype によっても solver 選択が分岐する（FP32・FP16・BFP16 でそれぞれ異なるファミリーへルーティング）。

補充（fallback）: rocBLAS は多段実行時フォールバックを持つ。CK は dot4 非対応時の逐次積和経路を保持。Tensile は ISA 別能力ゲートと gfx900 専用 lazy-load カタログを提供。

配布（shipped artifacts）: ROCm 7.2 パッケージには gfx900 ターゲットのプリコンパイル済みカーネル（rocBLAS 128 ファイル）とアーキテクチャ固有のチューニングデータ（MIOpen Perf DB 169K 行）が含まれる。これらの指標で gfx900 は RDNA 3（gfx1100: rocBLAS 96 ファイル、Perf DB なし）および RDNA 2（gfx1030: rocBLAS 88 ファイル、Perf DB 111K 行）を上回る。これは残留コードではなく、アクティブなビルドパイプラインの出力である。

2026-03-14 追補: ソースを CIFS から NVMe ローカルへ移し、再ビルド時に非互換な MLIR/CK 要素を切り離すことで、ローカル Debug MIOpen はビルド・実行に成功した。一方で -S ConvMlirIgemmFwd 強制は通常の適用ガードをバイパスできても、その先で MIIR_INVALID_PARAM や gfx900 向け Perf DB 不在に突き当たる。

確認済みギャップ: INT8 での自然 solver 選択は ConvDirectNaiveConvFwd のみ観測された。naive カーネルは dot 系命令を使わず要素ごとの乗算・加算で動作 — 機能的には成立するが最大性能ではない。
